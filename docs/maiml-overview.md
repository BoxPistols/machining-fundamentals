# MaiML — 計測分析データ共通フォーマット 概観

最終更新: 2026-05-15

本書は **MaiML (Measurement Analysis Instrument Markup Language)** の仕様要点・設計思想・切削加工分野からの関連性・実用パターン・将来性を、本リポジトリ（machining-fundamentals）の学習者向けに整理した参照ドキュメント。MaiML 自体の **実装は本リポジトリの守備範囲外**。フォーマット理解と将来動向のキャッチアップを目的とする。

一次情報源:

- **JIS K 0200**: 計測分析装置の分析データ共通フォーマット (MaiML データフォーマット)
- **共通データフォーマット対応ガイドライン Version 1.0**（2024 年 5 月 20 日 / 一般社団法人 日本分析機器工業会 = JAIMA / 経済産業省 委託事業）
- **MaiML スキーマ Version 1.0**: `maiml.xsd` 他 10 個の XSD ファイル群
- **JAIMA 公式**: <https://www.jaima.or.jp/>

本書は上記ガイドラインの **要約と再構成** にすぎず、規範文書ではない。準拠実装を作る場合は必ず JIS K 0200 とガイドライン本体を参照すること。

## 1. MaiML とは

| 項目 | 値 |
|---|---|
| 正式名称 | Measurement Analysis Instrument Markup Language |
| 略称 | MaiML |
| 規格 | JIS K 0200 |
| 仕様バージョン | 1.0 |
| ベース技術 | XML 1.0 |
| 名前空間 | `http://www.maiml.org/schemas` |
| 拡張子 | `.maiml`、`.mai`（外部ファイル同梱の ZIP は `.maizip`、ISO/IEC 21320-1 準拠） |
| 発行 | 一般社団法人 日本分析機器工業会 (JAIMA) |
| 助成 | 経済産業省 委託事業 |
| ガイドライン公開 | 2024-05-20 |

**目的**: 計測分析（材料・物質・部品の物性や組成を計測する全プロセス）の **入力 / 過程 / 出力** を、ベンダー非依存の共通 XML フォーマットで記述すること。装置メーカーごとに異なるバイナリ・独自 XML を「共通言語」へ橋渡しし、データの **保存・流通・再利用・統合分析** を可能にする。

**位置付け**: MaiML 自体は計測分析データを **記述する言語** であり、処理を行うプログラミング言語ではない。生成・変換は各ベンダー / 利用者がコンバータを作る前提。

## 2. machining-fundamentals との関連

本リポジトリ machining-fundamentals は **切削加工の入門学習教材**。「計測分析装置」の話がなぜここに登場するか:

- **章で扱う物性値の出所** — 章本文で言及する材料硬度・引張強度・工具摩耗 (VB) 値・表面粗さ (Ra/Rz) などは、すべて計測分析装置の出力。MaiML はその出力データの記述形式を標準化する取り組み
- **トレーサビリティ** — 切削条件 (Vc / f / ap) と工具寿命の対応データを再現性ある形で残すには、計測値・装置・プロトコルをセットで記録する必要がある。MaiML が想定する `<protocol>` + `<data>` + `<eventLog>` の構成はそのまま生産現場のトレーサビリティ要請に対応する
- **JIS / ISO 動向の把握** — 学習者が将来的に研究機関 / 装置メーカー / 大手製造業のデータ基盤に触れる際、MaiML 系のフォーマットに遭遇する可能性がある。基礎概念を押さえておくと無駄になりにくい

逆に、本リポジトリの章で **MaiML を直接実装することはない**。本書は「外の世界でこういう取り組みがある」という参照リンクの位置付け。

## 3. 仕様の概要

### 3.1 ベースになる外部標準

MaiML は車輪の再発明を避け、既存の XML 系標準を組み合わせて構成される。

| 標準 | 役割 | MaiML 内での所在 |
|---|---|---|
| **XML 1.0** | 基盤フォーマット (テキスト・タグ階層) | ファイル全体 |
| **PNML** (Petri Net Markup Language) | 計測分析の工程フローをペトリネットでモデル化 | `<protocol>` 配下 |
| **XES** (Extensible Event Stream) | 計測実行時のイベントログ追跡 | `<eventLog>` 配下 |
| **XML Signature** (xmldsig) | 改ざん防止の電子署名 | ファイル全体・部分要素 |
| **XML Encryption** (xenc) | 機密性確保の部分暗号化 | 機密扱いする要素 |
| **RDF** | 用語間オントロジー記述 | 別途リンク |
| **UUID** | 各要素の一意性担保 | グローバル要素属性 |

### 3.2 ファイル階層構造（第 0 〜 4 階層）

```xml
<maiml version="1.0" xsi:type="maimlRootType">     ← 第0階層
  <document>...</document>                          ← 第1階層
  <protocol>...</protocol>                          ← 第1階層
  <data>...</data>                                  ← 第1階層
  <eventLog>...</eventLog>                          ← 第1階層
</maiml>
```

**第 1 階層 4 要素** の役割:

| 要素 | 担当 |
|---|---|
| `<document>` | 文書メタ情報 (作成日、作成者、所有者、タイトル等) |
| `<protocol>` | 計測手順・プロトコル定義（PNML ベース） |
| `<data>` | 計測結果の実データ (1D/2D/3D スペクトル等) |
| `<eventLog>` | 実行時イベント記録 (XES ベース) |

第 2 階層以下は `<method>` `<program>` `<instruction>` `<results>` `<event>` 等が並ぶ。詳細はガイドライン §4 / §9 参照。

### 3.3 汎用データコンテナ

実データ・メタデータを **タグ名で固定せず、汎用コンテナ + key 属性で表現** する設計。

| 要素 | 用途 |
|---|---|
| `<property>` | 名前付きプロパティ。`key` 属性で意味を識別 |
| `<content>` | プロパティの値本体 (xsi:type で型指定、`units` `formatString` で精度・単位) |
| `<uncertainty>` | 計測値の不確かさを記述 |

```xml
<property key="material.tensile_strength">
  <content xsi:type="decimalContentType" units="MPa">615.2</content>
  <uncertainty>...</uncertainty>
</property>
```

ベンダー独自の用語は **名前空間で区切って共存** できる。これにより同じ装置データを「ベンダー名で読みたい人」と「標準語彙で読みたい人」が同じファイルから取り出せる。

### 3.4 ルート型による 2 ユースケース

| `xsi:type` | 用途 | 必須子要素 |
|---|---|---|
| `maimlRootType` | **CASE-1**: 計測の全記録 (プロトコル + データ + ログ) | document / protocol / data / eventLog |
| `protocolFileRootType` | **CASE-2**: プロトコルのみ (実行レシピ配布) | document / protocol |

CASE-1 は「測定して結果を共有する」、CASE-2 は「測定方法を配布する (実データなし)」の使い分け。

## 4. 設計原則 8 点（ガイドライン §2.1 要約）

| # | 原則 | 要点 |
|---|---|---|
| 1 | データの可読性 | バイナリでなくテキスト (XML) で記述 |
| 2 | 規定フォーマットの利用 | XML / PNML / XES に準拠して既存ツールを活用 |
| 3 | データの構造化 | 計測装置の出力を全てサイバー空間 / データレイクに格納可能に |
| 4 | プロセスのモデル化 | ペトリネットで離散事象として工程記述 |
| 5 | 用語の名前空間管理 | 用語そのものは固定せず名前空間で区別 |
| 6 | 独立可用性 | 外部ファイル URI 参照可、部分秘匿可 |
| 7 | データの一意性 / 改ざん防止 | UUID + XML 署名 (ブロックチェーン様の親子チェーン) |
| 8 | 追跡可能性 | XES ベースのイベントログで実行追跡 |

特に **「ベンダー用語を MaiML 側で固定しない」** が大きな設計判断。AnIML 等の先行 XML 標準は装置別タグを定めるが、MaiML は名前空間に委ね、**進化する用語に長期追従する** ことを優先している。

## 5. 想定ユーザー（ガイドライン §1）

| ケース | 対象 | 切削加工分野での具体例 |
|---|---|---|
| **A** | 計測分析機器・ソフトウェアの開発 / 製造関係者 | 工具摩耗測定機ベンダー、3D 表面形状計測ソフト開発者 |
| **B** | 新規材料の開発 / 製造関係者 | 切削工具用超硬・サーメット・cBN の材料試験部門 |

本リポジトリの学習者がもっとも遭遇しやすいのは **ケース B (材料試験データ) と ケース C (試験片プロトコル)**。

## 6. 実用パターン例

ガイドライン §10 で示される利用シナリオから、本リポジトリの章テーマに関連するものを抜粋・翻案。

### 6.1 自社で作製した試料を外部機関で測定

切削工具メーカーが新組成超硬合金を試作 → 第三者機関で組成分析 → 結果を社内に取り込み比較。

- **MaiML が解く問題**: 第三者機関の出力フォーマットが装置依存 (Bruker / Thermo / 島津 等で別形式) のため、社内 DB 取り込みが手作業
- **MaiML 化後**: プロトコル・装置・結果が共通スキーマで届くため、自動取り込みと社内データレイクとの統合が成立

### 6.2 工具寿命試験データの社内・系列横断流通

複数工場 / 複数材料 / 複数工具で取った VB-時間曲線を、同一データレイクで集約・回帰分析。

- **MaiML が解く問題**: 工場 A は CSV、工場 B は Excel、工場 C は装置生 XML。回帰前のクレンジングに大半の工数
- **MaiML 化後**: `<property key="cutting.flank_wear_VB">` 形式で揃えて格納、SQL/PySpark から直叩き

### 6.3 加工後検査データのトレーサビリティ

航空機エンジン部品 (タービンディスク等) の旋削後寸法・粗さデータを、納入先で検証可能な形で保存。

- **MaiML が解く問題**: 納入後 10〜30 年の保管期間中に装置メーカーがフォーマットを廃止する可能性
- **MaiML 化後**: テキスト + XML 署名で長期保管、改ざん検出まで仕様内

### 6.4 AI/ML 学習用データセットの標準化

材料硬度や工具摩耗を入力に、最適切削条件を出力する機械学習モデルの教師データを多機関から集める。

- **MaiML が解く問題**: 機関ごとにラベル付けが違う、欠損項目の扱いが違う、再現条件が記述されていない
- **MaiML 化後**: プロトコル + 不確かさ込みで配信されるので、再現性ある学習データセットが構築可能

## 7. 関連標準とエコシステム

| 関連 | 内容 |
|---|---|
| **JIS K 0200:2024** | MaiML 本体規格。日本産業規格として制定済み |
| **PNML** | 国際規格 (ISO/IEC 15909-2)。`<protocol>` 内で工程フロー記述に使用 |
| **XES** | IEEE 1849-2016。`<eventLog>` 内のイベント記述に使用 |
| **AnIML** | ASTM E13.15 で策定中の類似目的フォーマット。MaiML はベンダー用語固定方針が異なる |
| **mzML / netCDF / HDF5** | 各分野固有のフォーマット。MaiML から外部ファイルとして引用可 |
| **ISO 化** | **ISO 23495** として ISO/TC 201 (Surface chemical analysis) / SC 10 配下で国際標準化が進行中。JIS K 0200:2024 制定済み・ISO 化はその国際展開フェーズに位置付けられる |

## 8. 切削加工分野からみた将来性

ガイドライン本体には「切削加工」の事例は明示されていないが、ケース B / C と §10 のユースケースから類推できる将来像:

1. **工具〜加工〜検査の縦断データ流通**: 工具メーカー(材料試験) → 工作機械(加工条件・モニタリング) → 部品メーカー(後検査) を一本のデータパスでつなげる
2. **ベンダーロックイン解消**: 計測装置の買い替え時に過去データの読み出しが切れない
3. **規格・監査対応の自動化**: 航空・医療・原子力など規格認証が重い領域で、トレーサビリティを満たす最小フォーマットとして採用される可能性
4. **AI 学習データ標準化**: 「同じ条件で測ったか」を機械可読に確認できるため、Garbage-in を減らす入口
5. **デジタルツインの素材**: 加工条件 + 検査結果のペアを大量に蓄えると、加工パラメータ→品質予測モデルの学習データになる

逆に普及のハードル:

- XML の verbose さ (HDF5 等のバイナリと比べて容量・速度で劣る)
- 既存装置メーカーが独自フォーマットから移行する動機が弱い
- 海外標準 (AnIML) との競合・棲み分け
- ツールチェーンが整備途上 (汎用ビューア・エディタが少ない)

これらは MaiML に限らず標準化フォーマット一般の課題であり、JIS 制定 + 国際展開でどこまで採用が広がるかが鍵。
